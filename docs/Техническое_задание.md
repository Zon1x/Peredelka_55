## 1. Общие сведения

### 1.1 Полное наименование системы

Веб-платформа (информационно-справочный корпоративный сайт) **«Peredelka55»** для размещения сведений о компании, услугах переоборудования коммерческого транспорта, портфолио выполненных работ, отзывах клиентов, акциях и приёме заявок с сайта.

### 1.2 Основание для разработки

Потребность заказчика в **представительстве в сети Интернет**, снижении нагрузки на телефонные консультации за счёт структурированного каталога услуг и **цифрового канала приёма заявок**.

### 1.3 Плановые сроки

Определяются календарным планом ВКР / договором с заказчиком.

### 1.4 Источники разработки

Анализ предметной области переоборудования спецтехники; обзор сайтов конкурентов; требования к безопасности веб-приложений при публикации в сети.

---

## 2. Назначение и цели создания системы

### 2.1 Назначение

Система предназначена для:

- информирования посетителей об услугах, сроках, гарантиях и ориентировочной стоимости;
- демонстрации портфолио с визуальным сравнением «до/после»;
- публикации отзывов после модерации;
- информирования об акциях и приёме заявок на участие (в т.ч. со ссылкой на отзыв на стороннем ресурсе);
- приёма обращений через формы и доставки уведомлений ответственным лицам по электронной почте.

### 2.2 Цели

- повышение узнаваемости бренда и доверия к компании;
- рост числа целевых обращений;
- централизация контента (услуги, проекты, акции) в единой админ-панели.

---

## 3. Характеристика объекта автоматизации

Объект — **деятельность компании** по консультированию и приёму заказов на переоборудование транспорта. Субъекты: **посетитель сайта**, **администратор контента**, **администратор сервера** (развёртывание).

## 4. Требования к системе

### 4.1 Требования к функциональным возможностям

**4.1.1 Подсистема публичного сайта**

| № | Требование |
|---|------------|
| Ф.1 | Отображение главной страницы с блоками: калькулятор ориентировочной стоимости, быстрая заявка, статистика компании, выборочное портфолио, отзывы, акции, услуги по категориям. |
| Ф.2 | Каталог услуг: список по всем категориям и фильтрация по выбранной категории; карточка услуги с полным описанием, ценой «от», сроками, гарантией, процессом работы. |
| Ф.3 | Портфолио: список проектов с фильтром по категории техники; детальная страница с двумя изображениями «до» и «после», описанием, типом шасси, датой завершения, связью с услугами. |
| Ф.4 | Отзывы: просмотр только опубликованных; форма отправки нового отзыва; новые отзывы не публикуются до установки признака публикации в админ-панели. |
| Ф.5 | Акции: список активных акций; детальная страница; форма заявки на бонус по акции (ФИО, телефон, URL отзыва, комментарий). |
| Ф.6 | Контакты: отображение реквизитов и контактов из настроек сайта; полная форма заявки (имя, телефон, email, услуга, сообщение). |
| Ф.7 | Страницы ошибок 404 и 500 с фирменным оформлением. |
| Ф.8 | Адаптивная вёрстка для основных разрешений мобильных устройств и настольных мониторов. |

**4.1.2 Подсистема администрирования (Django Admin)**

| № | Требование |
|---|------------|
| А.1 | Управление настройками сайта (контакты, реквизиты, SEO, соцсети) и показателями статистики на главной. |
| А.2 | CRUD по категориям и услугам, категориям и элементам портфолио, отзывам, заявкам, акциям и заявкам на бонус. |
| А.3 | Загрузка изображений для услуг и портфолио. |

**4.1.3 Калькулятор**

Расчёт ориентировочной суммы на сервере: `цена_от × коэффициент_сложности × количество`, при отсутствии цены — вывод без числовой суммы (поведение интерфейса согласовать с заказчиком; в текущей реализации отображается только при наличии `price_from`).

### 4.2 Требования к надёжности

- Сохранение заявок и отзывов в БД независимо от успешной отправки email.
- Логирование ошибок приложения (настройка через `LOGGING` в Django).

### 4.3 Требования к информационной безопасности

- Защита форм от **CSRF** (встроенные механизмы Django).
- Хранение `SECRET_KEY` и параметров БД в **переменных окружения**, не в системе контроля версий.
- Разграничение доступа: публичная часть только на чтение контента и приём форм; изменение данных — только через аутентифицированный доступ к `/admin/`.
- В продакшене `DEBUG=False`, корректные `ALLOWED_HOSTS` и `CSRF_TRUSTED_ORIGINS`.

### 4.4 Требования к составу и параметрам технических средств

**Сервер приложений:** Python 3.x, веб-сервер совместимый с WSGI (например, Gunicorn за обратным прокси).

**СУБД:** PostgreSQL (рекомендуется для эксплуатации) или SQLite (допускается для отладки).

**Клиент:** браузер с поддержкой HTML5/CSS3/ES6 (актуальные версии Chrome, Firefox, Edge, Safari).

**Зависимости проекта** (ориентир): Django 5.2.x, Pillow, python-dotenv, whitenoise, при необходимости redis/django-redis, gunicorn, psycopg2-binary, dj-database-url (см. `requirements.txt`).

### 4.5 Требования к информационной и программной совместимости

- Кодировка UTF-8, язык интерфейса русский.
- Раздача статики через WhiteNoise или веб-сервер; медиа — через настройку веб-сервера или облачное хранилище по решению эксплуатации.

### 4.6 Требования к маркировке и упаковке

Не применимо (программное изделие).

### 4.7 Требования к транспортированию и хранению

Репозиторий и резервные копии БД и медиа хранятся по политике заказчика.

### 4.8 Специальные требования

- Настройка SMTP для уведомлений о заявках (`EMAIL_*` / стандартные параметры Django в окружении).
- Опционально: интеграция карты проезда в шаблон контактов (Яндекс/Google) при наличии API-ключа и политики конфиденциальности.

## 6. Порядок контроля и приёмки

Приёмка по пунктам раздела 4.1 на тестовом стенде: успешное прохождение сценариев просмотра контента, отправки форм, модерации отзыва, получения (или имитации) email при настроенной почте, отсутствие критических ошибок в логах при типовой нагрузке.

Критерий отказа: невозможность сохранить заявку, утечка персональных данных из-за некорректной конфигурации, отказ публичной части при штатном заполнении контента.
